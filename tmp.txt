좋아.
지금까지 정리된 방향을 기준으로, **외부 승인 서비스(approval service)를 구현하기 위한 작업 명령서**를 바로 개발에 넘길 수 있는 형태로 작성해줄게.

아래 문서는 전제, 목표, 요구사항, API, 데이터 모델, Keycloak 연동 방식, 운영 고려사항까지 포함한 **구현 지시서** 형태야.

---

# 외부 승인 서비스 개발 작업 명령서

## 1. 작업 개요

Keycloak 로그인 과정에서 `terms` Required Action 이후 `approval` Required Action이 수행되도록 구성한다.
`approval` Required Action은 **외부 서비스를 실시간 조회하지 않고**, Keycloak 사용자 attribute 에 저장된 승인 상태만 확인하여 로그인 허용 여부를 결정한다.

따라서 외부 승인 서비스의 역할은 다음과 같다.

1. 특정 client 에 대한 사용자 승인 대기 목록을 관리한다.
2. 관리자가 사용자 승인/거절을 처리할 수 있어야 한다.
3. client 별 자동 승인 규칙(auto approve rule)을 관리할 수 있어야 한다.
4. 승인 결과를 Keycloak 사용자 attribute 에 반영한다.
5. Keycloak 로그인 경로에는 직접 개입하지 않는다.

---

## 2. 목표

외부 승인 서비스는 승인 정책의 원본 시스템(Source of Truth)으로 동작하되,
로그인 시 최종 판단은 Keycloak 내부 상태(user attribute)만 사용하도록 한다.

즉 구조는 아래와 같다.

```text
외부 승인 서비스
  - 승인 정책 관리
  - 자동 승인 규칙 관리
  - 관리자 승인/거절 처리
  - Keycloak 사용자 attribute 동기화

Keycloak
  - terms Required Action
  - approval Required Action
  - approval.<clientId> attribute 로 승인 여부 판단
```

---

## 3. Keycloak 연동 전제

### 3.1 승인 상태 저장 방식

Keycloak 사용자 attribute 에 아래 형식으로 저장한다.

```text
approval.<clientId> = APPROVED | PENDING | REJECTED
```

예시:

```text
approval.client-a = APPROVED
approval.client-b = PENDING
approval.client-c = REJECTED
```

### 3.2 승인 상태 의미

* `APPROVED`: 로그인 허용
* `PENDING`: 승인 대기 화면 표시
* `REJECTED`: 거절 안내 화면 표시

### 3.3 포털 URL

Keycloak client attribute 에 아래 값을 사용할 수 있다.

```text
approval_portal_url
```

이는 Keycloak approval 화면에서 “승인 포털로 이동” 버튼 링크로 사용된다.

## 4. 외부 승인 서비스의 핵심 책임

외부 승인 서비스는 최소한 아래 기능을 제공해야 한다.

### 4.1 승인 요청 관리

* 어떤 사용자가 어떤 client 에 대해 승인 대기 중인지 조회 가능해야 한다.
* 상태별 조회가 가능해야 한다.
* 대기 생성 시점, 승인 시점, 승인자 정보 등을 저장해야 한다.

### 4.2 관리자 승인/거절 처리

* 관리자는 특정 사용자 + 특정 client 조합에 대해 승인 또는 거절할 수 있어야 한다.
* 처리 결과는 서비스 내부 DB에 저장해야 한다.
* 처리 후 Keycloak 사용자 attribute 를 즉시 갱신해야 한다.

### 4.3 자동 승인 규칙 관리

* client 별로 auto approve rule 을 설정할 수 있어야 한다.
* 예: `department == "People"` 이면 자동 승인
* 규칙은 Keycloak SPI 내부가 아니라 외부 승인 서비스에서 관리한다.

### 4.4 Keycloak 상태 동기화

* 승인 결과를 Keycloak Admin API 로 사용자 attribute 에 반영해야 한다.
* 실시간 로그인 판정은 Keycloak attribute 를 기준으로만 수행되도록 보장해야 한다.

## 6. 권장 데이터 모델

## 6.1 approval_requests

사용자별 client 승인 상태 저장

주요 컬럼 예시:

* `id`
* `realm_name`
* `client_id`
* `user_id`  (Keycloak user id)
* `username`
* `email`
* `status` (`PENDING`, `APPROVED`, `REJECTED`)
* `requested_at`
* `decided_at`
* `decided_by`
* `decision_reason`
* `source` (`MANUAL`, `AUTO_RULE`, `SYNC`)
* `last_synced_at`
* `created_at`
* `updated_at`

### 제약

* `(realm_name, client_id, user_id)` unique

---

## 6.2 auto_approve_rules

client 별 자동 승인 규칙 관리

주요 컬럼 예시:

* `id`
* `realm_name`
* `client_id`
* `field_name`
* `operator`
* `expected_value`
* `enabled`
* `priority`
* `description`
* `created_at`
* `updated_at`

예시 데이터:

```text
realm_name = myrealm
client_id = client-a
field_name = department
operator = EQUALS
expected_value = People
enabled = true
priority = 100
```

### 비고

초기 버전에서는 연산자를 단순화한다.

권장 초기 지원 연산자:

* `EQUALS`
* `IN`
* `EXISTS`

복잡한 논리식은 초기 범위에서 제외한다.

---

## 6.3 approval_audit_logs

승인 처리 이력 저장

주요 컬럼 예시:

* `id`
* `realm_name`
* `client_id`
* `user_id`
* `action` (`REQUESTED`, `APPROVED`, `REJECTED`, `AUTO_APPROVED`, `SYNCED_TO_KEYCLOAK`)
* `actor_type` (`ADMIN`, `SYSTEM`)
* `actor_id`
* `before_status`
* `after_status`
* `message`
* `created_at`

## 7. Keycloak 동기화 규칙

## 7.1 attribute key 규칙

Keycloak 사용자 attribute key 는 반드시 아래 규칙을 따른다.

```text
approval.<clientId>
```

예시:

```text
approval.client-a
approval.hr-portal
approval.people-system
```

## 7.2 attribute value 규칙

값은 반드시 아래 중 하나만 사용한다.

```text
APPROVED
PENDING
REJECTED
```

소문자/혼합형 사용 금지.

## 8. 권장 동작 흐름

## 8.1 승인 대기 생성 흐름

1. 특정 사용자에게 client 접근 요청이 발생한다.
2. 외부 승인 서비스는 해당 사용자/client 조합의 현재 상태를 조회한다.
3. auto approve rule 을 평가한다.
4. 조건 충족 시 `APPROVED`
5. 조건 미충족 시 `PENDING`
6. 결과를 DB에 저장한다.
7. 결과를 Keycloak user attribute 로 동기화한다.

---

## 8.2 관리자 승인 흐름

1. 관리자가 승인 포털에서 대기 목록을 조회한다.
2. 특정 사용자/client 조합을 승인한다.
3. 외부 승인 서비스는 DB 상태를 `APPROVED` 로 변경한다.
4. 감사 로그를 남긴다.
5. Keycloak 사용자 attribute `approval.<clientId>` 를 `APPROVED` 로 갱신한다.

---

## 8.3 관리자 거절 흐름

1. 관리자가 특정 요청을 거절한다.
2. 상태를 `REJECTED` 로 변경한다.
3. 감사 로그를 남긴다.
4. Keycloak 사용자 attribute 를 `REJECTED` 로 갱신한다.

---

## 9. API 요구사항

아래 API 는 최소 구현 범위다.

## 9.1 승인 요청 생성 또는 상태 보장

`POST /api/v1/approval-requests/ensure`

목적:

* 사용자/client 조합의 승인 상태를 생성 또는 보장
* auto approve rule 평가 포함 가능

요청 예시:

```json
{
  "realmName": "myrealm",
  "clientId": "client-a",
  "user": {
    "id": "keycloak-user-id",
    "username": "hong",
    "email": "hong@example.com",
    "attributes": {
      "department": ["People"],
      "division": ["HR"]
    }
  }
}
```

응답 예시:

```json
{
  "realmName": "myrealm",
  "clientId": "client-a",
  "userId": "keycloak-user-id",
  "status": "APPROVED",
  "source": "AUTO_RULE"
}
```

### 비고

이 API 는 초기 사용자 등록, 배치, 사전 동기화 등에 사용할 수 있다.
Keycloak 로그인 중 실시간 호출 용도로 사용하지 않는다.

---

## 9.2 승인 대기 목록 조회

`GET /api/v1/approval-requests?realmName=myrealm&clientId=client-a&status=PENDING`

목적:

* 관리자가 특정 client 의 대기 사용자 목록을 조회

응답 예시:

```json
{
  "items": [
    {
      "userId": "u1",
      "username": "hong",
      "email": "hong@example.com",
      "status": "PENDING",
      "requestedAt": "2026-03-06T10:00:00Z"
    }
  ],
  "total": 1
}
```

---

## 9.3 승인 처리

`POST /api/v1/approval-requests/{id}/approve`

요청 예시:

```json
{
  "approvedBy": "admin-user-id",
  "reason": "부서 확인 완료"
}
```

동작:

* DB 상태 `APPROVED`
* 감사 로그 저장
* Keycloak attribute 동기화

---

## 9.4 거절 처리

`POST /api/v1/approval-requests/{id}/reject`

요청 예시:

```json
{
  "rejectedBy": "admin-user-id",
  "reason": "접근 권한 없음"
}
```

동작:

* DB 상태 `REJECTED`
* 감사 로그 저장
* Keycloak attribute 동기화

---

## 9.5 자동 승인 규칙 CRUD

### 규칙 생성

`POST /api/v1/auto-approve-rules`

요청 예시:

```json
{
  "realmName": "myrealm",
  "clientId": "client-a",
  "fieldName": "department",
  "operator": "EQUALS",
  "expectedValue": "People",
  "enabled": true,
  "priority": 100,
  "description": "People 부서는 자동 승인"
}
```

### 규칙 조회

`GET /api/v1/auto-approve-rules?realmName=myrealm&clientId=client-a`

### 규칙 수정

`PATCH /api/v1/auto-approve-rules/{id}`

### 규칙 비활성화

`POST /api/v1/auto-approve-rules/{id}/disable`

## 10. 자동 승인 규칙 엔진 요구사항

초기 버전은 단순 규칙만 지원한다.

### 입력

* `realm_name`
* `client_id`
* 사용자 기본 필드
* 사용자 attribute

### 평가 대상 예시

* `username`
* `email`
* `department`
* `division`

### 초기 연산자

* `EQUALS`
* `IN`
* `EXISTS`

### 평가 결과

* 하나라도 승인 규칙에 매칭되면 `APPROVED`
* 매칭되지 않으면 `PENDING`

### 주의

규칙 충돌 처리 방식은 초기 버전에서 단순화한다.

권장:

* 높은 priority 먼저 평가
* 첫 매칭 즉시 종료

## 11. Keycloak 연동 요구사항

## 11.1 Keycloak Admin API 사용

외부 승인 서비스는 Keycloak Admin API 를 통해 사용자 attribute 를 갱신해야 한다.

갱신 대상:

```json
{
  "attributes": {
    "approval.client-a": ["APPROVED"]
  }
}
```

### 주의

기존 사용자 attribute 전체를 덮어쓰지 않도록 병합 방식으로 업데이트해야 한다.

---

## 11.2 동기화 실패 처리

Keycloak 동기화 실패 시:

1. DB 상태 변경은 저장
2. 동기화 실패 로그 기록
3. 재시도 대상 큐 또는 테이블에 적재
4. 운영자 확인 가능해야 함

즉, 승인 처리와 Keycloak 동기화는 논리적으로 분리하되,
최종 일관성(Eventual Consistency)을 보장해야 한다.

---

## 12. 비기능 요구사항

## 12.1 안정성

* Keycloak 로그인 경로에 직접 의존하지 않도록 한다.
* 외부 승인 서비스 장애가 로그인 장애로 전이되지 않아야 한다.

## 12.2 감사 가능성

* 누가 언제 누구를 어떤 client 에 대해 승인/거절했는지 기록해야 한다.

## 12.3 추적 가능성

* 특정 사용자의 승인 상태를 DB와 Keycloak 양쪽에서 확인 가능해야 한다.

## 12.4 멱등성

* 승인 API, 거절 API, Keycloak 동기화 API 는 재시도 가능하도록 멱등성을 고려한다.

---

## 13. 운영 정책

## 13.1 상태 원본

승인 정책의 원본은 외부 승인 서비스 DB다.

## 13.2 로그인 판정 원본

로그인 시 최종 판정 원본은 Keycloak user attribute 다.

## 13.3 불일치 발생 시 처리

DB 상태와 Keycloak attribute 값이 다를 수 있으므로, 아래 기능이 필요하다.

* 특정 사용자/client 상태 재동기화 API
* 전체 재동기화 배치 또는 작업자
* 불일치 탐지 로그/모니터링

## 15. 개발 시 주의사항

### 15.1 실시간 로그인 연동 금지

외부 승인 서비스는 Keycloak 로그인 시 직접 호출되는 구조로 만들지 않는다.

### 15.2 승인 상태와 권한 분리

승인 상태는 user attribute 로 관리하고, 권한(role)과 혼합하지 않는다.

### 15.3 clientId 안정성 보장

attribute key 가 `approval.<clientId>` 형식이므로 clientId 는 운영 중 자주 변경하지 않는 것을 전제로 한다.

### 15.4 문자열 상수 표준화

상태값은 반드시 아래 상수만 사용한다.

```text
APPROVED
PENDING
REJECTED
```
